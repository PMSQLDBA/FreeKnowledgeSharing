## STEP 1: Verify Current Certificate Status

**Time**: 2 minutes

```sql
-- Run this BEFORE starting renewal
USE master

-- Check current certificate
SELECT 
    name AS CertificateName,
    expiry_date AS ExpiryDate,
    DATEDIFF(DAY, GETDATE(), expiry_date) AS DaysRemaining
FROM sys.certificates
WHERE name = 'Karthikeya_TDE_Cert'

-- Expected result:
-- Karthikeya_TDE_Cert | 2027-07-18 | 1-7 days remaining

-- If result shows:
-- - 0 rows: Certificate missing (EMERGENCY!)
-- - Negative days: Certificate EXPIRED (EMERGENCY!)
-- - Positive days: Good, continue with renewal
```

**If certificate is missing or expired**: STOP and see Emergency Procedures section

## STEP 3: Backup New Certificate IMMEDIATELY

**Time**: 2 minutes

**CRITICAL**: Backup BEFORE switching DEK!

```sql
-- Run this in master database
USE master

-- Backup new certificate with private key
BACKUP CERTIFICATE [Karthikeya_TDE_Cert_New]
TO FILE = 'E:\path\TDE_Testing\Karthikeya_TDE_Cert_New.cer'
WITH PRIVATE KEY
(
    FILE = 'E:\path\TDE_Testing\Karthikeya_TDE_Key_New.pvk',
    ENCRYPTION BY PASSWORD = 'YourKeyPassword123'  -- IMPORTANT: Use CORRECT password
)

-- Expected message:
-- "Certificate backed up successfully"

-- Verify backup files were created
-- Use Windows File Explorer to check:
-- E:\path\TDE_Testing\Karthikeya_TDE_Cert_New.cer   (should exist)
-- E:\path\TDE_Testing\Karthikeya_TDE_Key_New.pvk    (should exist)
```

**If backup fails**:
```
Error: "An error occurred while generating a key..."
Solution: Check password is correct, try again
```

## STEP 4: Switch DEK to New Certificate

**Time**: 5 minutes (this is the "downtime" moment)

**IMPORTANT**: New connections will need brief reconnection

```sql
-- Run this in Karthikeya database
USE [Karthikeya]

-- Switch DEK to new certificate
ALTER DATABASE ENCRYPTION KEY
ENCRYPTION BY SERVER CERTIFICATE [Karthikeya_TDE_Cert_New]

-- Expected message:
-- "Query executed successfully"

-- This operation:
-- - Takes 5-10 minutes on large databases
-- - Causes new connections to briefly pause
-- - Existing connections continue working
-- - No data is lost or corrupted
```

## STEP 5: Verify New Certificate is Protecting Database

**Time**: 2 minutes

```sql
-- Run this in master database
USE master

-- Verify DEK is now protected by new certificate
SELECT
    db.name AS DatabaseName,
    dek.encryption_state,
    CASE dek.encryption_state WHEN 3 THEN 'ENCRYPTED' ELSE 'NOT ENCRYPTED' END AS Status,
    c.name AS CertificateName,
    c.expiry_date AS ExpiryDate,
    DATEDIFF(DAY, GETDATE(), c.expiry_date) AS DaysRemaining
FROM sys.dm_database_encryption_keys dek
INNER JOIN sys.databases db ON dek.database_id = db.database_id
LEFT JOIN sys.certificates c ON dek.encryptor_thumbprint = c.thumbprint
WHERE db.name = 'Karthikeya'

-- Expected result:
-- Karthikeya | 3 | ENCRYPTED | Karthikeya_TDE_Cert_New | 2028-07-18 | 365
```

**If certificate name is still old**:
```
CertificateName: Karthikeya_TDE_Cert (not _New)
Problem: DEK switch didn't work
Solution: Run Step 4 again or check for errors
```

## Troubleshooting Common Issues

### Issue 1: "Certificate already exists"

```
Error: "Certificate with name 'Karthikeya_TDE_Cert_New' already exists"
Cause: Tried to create same certificate twice
Solution: 
  1. Drop old new certificate: DROP CERTIFICATE [Karthikeya_TDE_Cert_New]
  2. Create again with different name or after cleanup
  3. OR: Use existing new certificate and continue from Step 4
```

### Issue 2: "Cannot backup certificate"

```
Error: "An error occurred while generating a key required by this operation (15508)"
Cause: Wrong password or certificate malformed
Solution:
  1. Verify password is correct
  2. If password wrong: Recreate certificate with correct password
  3. If certificate malformed: Drop and recreate
```

### Issue 3: "Cannot switch DEK"

```
Error: "Certificate not found" when switching DEK
Cause: Certificate doesn't exist or wrong name
Solution:
  1. Verify new certificate exists: SELECT * FROM sys.certificates WHERE name = 'Karthikeya_TDE_Cert_New'
  2. If missing: Go back to Step 2, create it again
  3. If exists: Try DEK switch again
```

### Issue 4: "Database still shows old certificate"

```
Problem: Query shows DEK still protected by old certificate
Cause: DEK switch didn't complete
Solution:
  1. Wait 5-10 minutes for async completion
  2. Run verification query again
  3. If still showing old: Run ALTER DATABASE ENCRYPTION KEY again
  4. Check for errors in SQL Server error log
```

## Rollback Plan (If Something Goes Wrong)

### If Renewal Fails Before DEK Switch

```
Situation: Created new certificate but didn't switch DEK
Status: Database still protected by old certificate (safe!)
Action:
  1. Drop new certificate: DROP CERTIFICATE [Karthikeya_TDE_Cert_New]
  2. Investigate error
  3. Fix issue
  4. Start renewal again from Step 2
Impact: Zero - database unaffected
```

### If Renewal Fails After DEK Switch

```
Situation: DEK switched to new certificate but something wrong
Status: Database encrypted with new certificate (may have issues)
Action:
  1. Try to access database - see if it works
  2. If works: Everything fine, continue
  3. If doesn't work: Switch DEK back to old certificate
     ALTER DATABASE ENCRYPTION KEY
     ENCRYPTION BY SERVER CERTIFICATE [Karthikeya_TDE_Cert]
  4. Investigate and retry renewal
Impact: Minimal - can revert to old certificate
```

### If Certificate Backup Failed

```
Situation: New certificate created but backup failed
Status: Database encrypted but no backup files
Action:
  1. URGENT: Backup new certificate immediately
     BACKUP CERTIFICATE [Karthikeya_TDE_Cert_New] TO FILE...
  2. If still fails: Restore old certificate, retry renewal
  3. Ensure backup succeeds before continuing
Impact: High - must have backup files!
```
